# uv tooling migration implementation plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate the dev workflow to uv — dev dependencies in `pyproject.toml` `[dependency-groups]`, a committed `uv.lock`, and CI plus README using uv — removing the pip-tools `requirements.ini`/`requirements.txt` flow.

**Architecture:** uv manages the virtual environment and a universal lockfile from `pyproject.toml`. Runtime dependencies stay in `[project.dependencies]`; the dev tools (black, coverage, twine) move to a PEP 735 `dev` dependency group. CI installs uv, runs `uv sync --locked`, and invokes tools through `uv run`; the publish job builds with `uv build` and uploads with `uvx twine`. The setuptools + setuptools_scm build backend is unchanged, so the version still comes from git tags.

**Tech Stack:** uv, PEP 735 dependency groups, setuptools + setuptools_scm (unchanged), CircleCI.

## Task 2: Switch CI to uv

**Files:**
- Modify: `.circleci/config.yml`

- [ ] **Step 1: Replace `.circleci/config.yml` with the uv-based pipeline**

```yaml
version: 2.1
_definitions:
  - &semver_pattern /^\d+\.\d+\.\d+.*/
executors:
  python:
    docker:
      - image: cimg/python:3.12.1
jobs:
  test-python:
    executor: python
    steps:
      - checkout
      - run:
          name: Install uv
          command: pip install uv
      - run:
          name: Install dependencies
          command: uv sync --locked
      - run:
          name: Check style
          command: uv run black --check .
      - run:
          name: Run tests
          command: |
            uv run coverage run --source jinjabread -m unittest discover .
            uv run coverage report
            uv run coverage html
      - store_artifacts:
          path: htmlcov
  publish-python-package:
    executor: python
    steps:
      - checkout
      - run:
          name: Install uv
          command: pip install uv
      - run:
          name: Create build
          command: uv build
      - run:
          name: Upload build
          command: uvx twine upload dist/*
workflows:
  main:
    jobs:
      - test-python:
          filters:
            tags:
              only: *semver_pattern
      - publish-python-package:
          context:
            - jinjabread-prod
          requires:
            - test-python
          filters:
            tags:
              only: *semver_pattern
            branches:
              ignore: /.*/
```

Notes: the publish job deliberately does not `uv sync` — `uv build` fetches the build backend on its own, and `uvx twine` runs twine ephemerally while still reading `TWINE_USERNAME`/`TWINE_PASSWORD` from the `jinjabread-prod` context, so the release auth mechanism is unchanged.

- [ ] **Step 2: Verify the pip-tools pipeline is fully replaced**

Run: `grep -n "pip-sync\|pip-tools\|venv/bin/activate" .circleci/config.yml`
Expected: no matches. (CircleCI validates the YAML itself on push, which is the maintainer's step; there is no project-local YAML parser to lean on.)

- [ ] **Step 3: Commit**

```bash
git add .circleci/config.yml
git commit -F - <<'EOF'
Run CI on uv instead of pip-tools

CI installs uv, restores the locked environment with `uv sync --locked`, and
runs black/coverage through `uv run`. The publish job builds with `uv build`
and uploads with `uvx twine`, keeping the existing PyPI token auth from the
jinjabread-prod context.
EOF
```
